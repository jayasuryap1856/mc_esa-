32-bit Addition using MSP430 Assembly
May 2023 · Dec 2023 · Jan-May 2024 4–6 marks
▾
// Question
Write MSP430 assembly to add two 32-bit hex numbers at memory locations 0x0220 and 0x0224, storing the 32-bit result in 0x0228. Use absolute/symbolic addressing. Do not modify 0x0220 and 0x0224.
// Answer

A 32-bit number spans two 16-bit words. Low word at base address, high word at base+2. We use ADD for the low 16 bits and ADDC (Add with Carry) for the high 16 bits to propagate the carry.

; Load low 16 bits of first operand (0x0220) into R4
MOV.W   &0x0220, R4
; Load low 16 bits of second operand (0x0224) into R5
MOV.W   &0x0224, R5
; Add low words — carry flag set if result overflows 16 bits
ADD.W   R5, R4
; Store low 16 bits of result
MOV.W   R4, &0x0228

; Load high 16 bits of first operand (0x0222)
MOV.W   &0x0222, R4
; Load high 16 bits of second operand (0x0226)
MOV.W   &0x0226, R5
; Add high words + carry from previous ADD
ADDC.W  R5, R4
; Store high 16 bits of result
MOV.W   R4, &0x022A
Key Points
ADD sets carry; ADDC adds carry into the high word — this is the standard MSP430 multi-precision addition pattern. Jan-May 2024 variant asks for a 33-bit result: store the final carry in 0x022C using MOV.W #0, R6 / ADDC.W #0, R6 / MOV.W R6, &0x022C.

Q03
MULT Subroutine + Compute Z = X*2 + Y*4
May 2023 · Jan-May 2024 8 marks
▾
// Question
1) Write a subroutine _MULT that multiplies R10 × R11, result in R10. All other registers unchanged.
2) Load X (Data_X) into R7 and Y (Data_Y) into R8.
3) Compute Z = X*2 + Y*4, save in R13. (X, Y < 100, no overflow)
// Answer

MSP430 has no hardware MUL instruction in the base architecture — multiply is done by repeated addition. We must save/restore registers to keep them unchanged.

; ========== _MULT Subroutine ==========
; Input:  R10 (multiplicand), R11 (multiplier)
; Output: R10 = R10 * R11
; Preserves: R11, R12 (used internally, saved via stack)
_MULT:
    PUSH    R11         ; save R11
    PUSH    R12         ; save R12
    MOV     R10, R12    ; R12 = original multiplicand (addend)
    CLR     R10         ; R10 = accumulator = 0
MULT_LOOP:
    CMP     #0, R11     ; if multiplier = 0, done
    JZ      MULT_DONE
    ADD     R12, R10    ; accumulator += multiplicand
    DEC     R11         ; multiplier--
    JMP     MULT_LOOP
MULT_DONE:
    POP     R12
    POP     R11
    RET

; ========== Main Code ==========
    ; Load X and Y
    MOV.W   &Data_X, R7    ; R7 = X
    MOV.W   &Data_Y, R8    ; R8 = Y

    ; Compute X*2
    MOV     R7, R10
    MOV     #2, R11
    CALL    #_MULT          ; R10 = X*2
    MOV     R10, R13        ; R13 = X*2 (partial Z)

    ; Compute Y*4
    MOV     R8, R10
    MOV     #4, R11
    CALL    #_MULT          ; R10 = Y*4

    ; Z = X*2 + Y*4
    ADD     R10, R13        ; R13 = Z
Tip
Since X and Y are small (<100), the repeated-addition loop is fast and no overflow occurs in 16-bit registers.
Q04
2×2 Keypad + LED Toggle (AppPortInit, keypadRead, toggleLED)
May 2023 · Dec 2023 8–10 marks
▾
// Question
2×2 keypad with rows R1,R2 and columns C1,C2. When key 1 is pressed, toggle LED (initially OFF).
1) Show hardware connection and justify input choice.
2) Write AppPortInit() — initialise ports.
3) Write keypadRead() — returns 0x01 for key1, 0x02 for key2…, 0xFF if none pressed.
4) Write toggleLED().
// Answer

Hardware Connection: Columns C1, C2 → P2.0, P2.1 (outputs, driven low one at a time). Rows R1, R2 → P2.2, P2.3 (inputs with pull-up resistors). LED on P1.0 (output). Pull-ups on rows ensure a HIGH when no key is pressed; a key press pulls the row LOW — easy to detect.

#include "msp430g2553.h"

// LED on P1.0, Columns on P2.0-P2.1, Rows on P2.2-P2.3

void AppPortInit(void) {
    // LED as output, initially OFF
    P1DIR |= BIT0;
    P1OUT &= ~BIT0;

    // Columns P2.0, P2.1 as outputs
    P2DIR |= (BIT0 | BIT1);
    P2OUT |= (BIT0 | BIT1);   // drive HIGH initially

    // Rows P2.2, P2.3 as inputs with pull-up resistors
    P2DIR &= ~(BIT2 | BIT3);
    P2REN |= (BIT2 | BIT3);   // enable pull resistors
    P2OUT |= (BIT2 | BIT3);   // select pull-UP
}

unsigned char keypadRead(void) {
    unsigned char key = 0xFF; // default: no key

    // Drive Column 1 LOW, Column 2 HIGH
    P2OUT &= ~BIT0;
    P2OUT |=  BIT1;
    __delay_cycles(100);  // settle time

    if (!(P2IN & BIT2)) key = 0x01;  // Key 1 (R1,C1)
    if (!(P2IN & BIT3)) key = 0x03;  // Key 3 (R2,C1)

    // Drive Column 2 LOW, Column 1 HIGH
    P2OUT |=  BIT0;
    P2OUT &= ~BIT1;
    __delay_cycles(100);

    if (!(P2IN & BIT2)) key = 0x02;  // Key 2 (R1,C2)
    if (!(P2IN & BIT3)) key = 0x04;  // Key 4 (R2,C2)

    // Restore both columns HIGH
    P2OUT |= (BIT0 | BIT1);
    return key;
}

void toggleLED(void) {
    P1OUT ^= BIT0;  // XOR toggles BIT0
}

int main(void) {
    WDTCTL = WDTPW + WDTHOLD;
    AppPortInit();
    while (1) {
        if (keypadRead() == 0x01) {
            toggleLED();
            __delay_cycles(200000); // debounce
        }
    }
}
Q05
Door Sensor + Buzzer using Timer A (100 Hz square wave)
May 2023 · Dec 2023 10 marks
▾
// Question
Door sensor on P1.1 (1 = open, 0 = closed). Buzzer on P1.7. Generate 100Hz square pulse on buzzer when door is open. Use Timer A interrupt (NOT PWM). Show: a) Port init b) Timer A config (mode, CCR, count) c) Start/stop logic d) ISR.
// Answer

Timer Calculation: SMCLK = 1MHz, 100Hz → period = 10ms → toggle every 5ms → CCR0 = 5000 − 1 = 4999. Using Up mode, ISR fires every 5ms and toggles buzzer → 100Hz square wave.

#include "msp430g2553.h"

void main(void) {
    WDTCTL = WDTPW + WDTHOLD;

    // a) Port Initialisation
    P1DIR &= ~BIT1;   // P1.1 = door sensor INPUT
    P1DIR |=  BIT7;   // P1.7 = buzzer OUTPUT
    P1OUT &= ~BIT7;   // buzzer OFF initially

    // b) Timer A0 Config (Up mode, SMCLK, CCR0=4999 → 5ms)
    TA0CCR0  = 4999;                     // 5ms at 1MHz
    TA0CCTL0 = CCIE;                     // enable CCR0 interrupt
    // Timer NOT started yet (MC bits = 00 by default)
    TA0CTL   = TASSEL_2 | ID_0;         // SMCLK, /1, timer stopped

    __bis_SR_register(GIE);             // enable global interrupts

    // c) Main loop: start/stop timer based on door
    while (1) {
        if (P1IN & BIT1) {
            // Door OPEN → start timer (Up mode)
            TA0CTL |= MC_1;
        } else {
            // Door CLOSED → stop timer, turn buzzer OFF
            TA0CTL &= ~MC_3;    // MC = 00 (stop)
            P1OUT  &= ~BIT7;    // ensure buzzer off
        }
    }
}

// d) Timer A0 ISR — fires every 5ms
#pragma vector = TIMER0_A0_VECTOR
__interrupt void Timer0_A0_ISR(void) {
    P1OUT ^= BIT7;   // toggle buzzer → 100Hz square wave
}
Q06
WDT LED Toggle — delay_func Analysis
May 2023 · Dec 2023 · Jan-May 2024 5–6 marks
▾
// Question
WDT enabled with SMCLK 1MHz, default count 32768. LED toggled in while loop using delay_func() (5001-iteration empty for loop). 1) What decides whether LED toggles at desired rate? 2) If not, how to fix it?
// Answer

1) What decides the toggle rate?

The LED toggle rate is determined by which expires first: the WDT timeout or the delay_func() execution time.

  • WDT timeout = 32768 / 1MHz = 32.768 ms
  • delay_func loop: 5001 iterations. Each iteration ≈ 2–3 cycles at 1MHz ≈ 10–15 ms
  • Total loop (toggle + delay) ≈ ~15ms < 32.768ms → WDT resets the CPU before the desired delay completes

So the LED doesn't toggle correctly — the CPU keeps resetting.

2) Fix: The WDT interrupt handler must periodically kick (reset) the WDT counter, OR the WDT should be re-enabled in the ISR. The fix is to add WDT reset inside the ISR:

// Fix 1: Kick WDT in ISR to prevent reset
#pragma vector = WDT_VECTOR
__interrupt void WDT_ISR(void) {
    WDTCTL = WDTPW + WDTCNTCL;  // reset WDT counter
}

// Fix 2: Alternatively, disable WDT and use it only as interval timer
// Change: WDTCTL = WDTPW + WDTHOLD;  (just disable it)
// And use a simple __delay_cycles() for the delay
Root Cause Summary
If delay_func() takes longer than 32.768ms, WDT fires first and resets the CPU. The code enables WDTIE and GIE, so WDT ISR fires — but without kicking the WDT counter, a subsequent timeout causes a PUC (reset).
Q07
Peripheral Polling Bug Analysis (MOV.B, CMP, BNE code)
May 2023 · Jan-May 2024 8 marks
▾
// Question
Evaluate the following code (correct or incorrect with reason):

1   MOV.B 0(R6), R4   ; load state (16-bit) into R4
5   CMP R4, #30
6   BNE CMP
8   MOV &2(R6), R8  ; load data byte to R8

1) Line 1 loads status of new data arrival properly
2) Line 5 gives assembler error
3) Lines 1,5,6 loop to check the status register for new data
4) Line 8 loads the byte value to R8
// Answer
  1. Line 1 — INCORRECT. MOV.B loads only 8 bits (.B = byte), but the status register is 16-bit. Should be MOV.W 0(R6), R4 to correctly read the 16-bit status value.
  2. Line 5 — INCORRECT (assembler error). CMP R4, #30 — in MSP430 syntax, the immediate operand must be the source and the register is the destination: CMP #30, R4 is correct. As written, #30 is in the destination field which is invalid for CMP.
  3. Lines 1, 5, 6 — INCORRECT for looping. The loop jumps back to label CMP (line 5), but line 1 (the MOV that reads the peripheral register) is NOT inside the loop. The status is read only once. To properly poll, line 1 must also be inside the loop — the jump should go back to a label before line 1.
  4. Line 8 — INCORRECT. MOV &2(R6), R8 uses symbolic (absolute) addressing & with an offset, which is invalid syntax. The correct form to access R6+2 is MOV.B 2(R6), R8 (register-indexed addressing, no &).
Corrected Code
POLL:
    MOV.W  0(R6), R4    ; correctly read 16-bit status
    CMP    #30, R4      ; compare with 30d
    JNE    POLL         ; loop if not equal
    MOV.B  2(R6), R8    ; load byte from R6+2

// Question

DEC.W R7 and JNZ loop_in follow.
1) What is the address of label loop_in?
2) How many clock cycles for all 3 instructions?
// Answer

1) Address of loop_in:

  • MOV.W #0xFF, R7 at 0x00C01A → opcode is 4037 00FF = 4 bytes
  • Next instruction (DEC.W R7) starts at 0x00C01A + 4 = 0x00C01E
  • loop_in label is at 0x00C01E

2) Clock cycles:

InstructionCyclesReason
MOV.W #0xFF, R72Immediate → register (constant generator if fits, else 2-word = 2 cycles)
DEC.W R71Uses constant generator SUB(#1, R7) = 1 cycle
JNZ loop_in2Jump instruction = 2 cycles (taken or not)

Total = 2 + 1 + 2 = 5 clock cycles for one pass through (first iteration).

Q09
ADC Resolution Bits + AD7641 Digital Output
May 2023 · Jan-May 2024 6–8 marks
▾
// Question
1) Sound 0–2.5V, resolution 20μV. How many bits ADC needed?
2) AD7641 18-bit ADC, 0–2.5V range, 1.5 MSPS. Input = 1.25V. What is digital output in hex? Can it record and reproduce 20kHz sound? Justify.
// Answer

1) Number of bits required:

Resolution = Vrange / 2ⁿ → 20×10⁻⁶ = 2.5 / 2ⁿ

2ⁿ = 2.5 / 20×10⁻⁶ = 125,000 → n = log₂(125000) ≈ 16.93 → n = 17 bits minimum

2a) Digital output for 1.25V:

18-bit ADC, full range = 2.5V, full scale = 2¹⁸ − 1 = 262143

Digital code = (Vin / Vref) × 2ⁿ = (1.25 / 2.5) × 262144 = 0.5 × 262144 = 131072 = 0x20000

2b) Can it record/reproduce 20kHz sound?

By Nyquist theorem, sampling rate must be ≥ 2 × signal frequency = 2 × 20kHz = 40kHz.

AD7641 samples at 1.5 MSPS which is far greater than 40kHz. → Yes, it can record and reproduce 20kHz sound.

Q10
Servo PWM — CCR Counts, Increment, Output Mode, Code Loop
May 2023 · Jan-May 2024 7–9 marks
▾
// Question
Servo: −90° to +90° in 10° steps, stop 1s each. PWM freq 50Hz. −90°=1ms, +90°=2ms. Duty proportional to angle. SMCLK=8MHz, Up/Down mode only.
1) CCR counts for −90° and +90°
2) Increment per 10°
3) Output mode
4) Code loop snippet (no init)
// Answer

1) CCR Counts:

Up/Down mode: period = 2 × CCR0 clock cycles. 50Hz → period = 20ms.

CCR0 = (SMCLK × period/2) = 8MHz × 10ms = 80,000

Pulse width → CCR1 count = pulse_width × SMCLK

  • −90°: 1ms pulse → CCR1 = 1×10⁻³ × 8×10⁶ = 8000
  • +90°: 2ms pulse → CCR1 = 2×10⁻³ × 8×10⁶ = 16000

2) Increment per 10°:

Total range = 180°, CCR1 range = 16000 − 8000 = 8000 counts

Per degree = 8000/180 ≈ 44.4 counts. Per 10° = ~444 counts

3) Output Mode: Reset/Set (Mode 7) — output goes HIGH when TAR counts up to TA0CCR1, LOW when counting back down past CCR1. This generates the PWM pulse.

4) Code Loop:

int ccr1 = 8000;       // start at -90 degrees
int step = 444;        // counts per 10 degrees

while (1) {
    TA0CCR1 = ccr1;    // set pulse width
    Delay_1s();        // wait 1 second at this position
    ccr1 += step;      // increment to next angle
    if (ccr1 > 16000) break; // stop at +90 degrees
}

UART Signal Analysis — Baud Rate, Byte, Parity
May 2023 · Dec 2023 · Jan-May 2024 6 marks
▾
// Question
UART: 8 bits/char, even parity (9th bit), 1 stop bit. Waveform shows bit period = 1ms, signal goes: START(0), then bit sequence, then STOP(1). Find: 1) Baud rate 2) Byte received 3) Parity bit — is there an error?
// Answer

1) Baud Rate:

Bit period = 1ms → Baud rate = 1/1ms = 1000 baud

2) Byte Received:

UART transmits LSB first after the start bit. Reading the waveform (start bit = LOW, then 8 data bits at 1ms each):

From the waveform pattern visible in the question (typical for this PYQ): bits are 1,1,0,1,0,1,0,0 (LSB first) → MSB first = 0,0,1,0,1,0,1,1 = 0x2B (43 decimal)

Note
Read your actual waveform carefully from the paper — the bit pattern depends on the exact signal levels. The method: identify start bit (HIGH→LOW transition), then read 8 bits at 1ms intervals, LSB first, reassemble MSB first.

3) Parity Bit:

Even parity: count number of 1s in the 8 data bits. If even, parity bit = 0; if odd, parity bit = 1.

Check the received parity bit (9th bit after data) against the computed parity. If they don't match → transmission error detected.

Q12
I2C vs UART — AT24C04 Memory + Gyro Chip Selection
May 2023 · Dec 2023 · Jan-May 2024 8 marks
▾
// Question
AT24C04 EEPROM (I2C addr: 1010000). Gyro/MEMS (I2C addr: 1101000, also supports UART at 9600bps). MSP430 must talk to both.
1) Advantage of I2C for both chips
2) Advantage of UART for Gyro + I2C for memory
3) Your selection for Gyro — I2C or UART? Justify. Draw connection diagram.
// Answer

1) Advantage of I2C for BOTH chips:

  • Only 2 wires (SDA, SCL) shared by both devices — saves GPIO pins
  • Both chips have unique I2C addresses → can be addressed individually on same bus
  • No chip select lines needed; addressing is done in protocol
  • Can support multiple slaves on same bus easily

2) Advantage of UART for Gyro + I2C for Memory:

  • UART is a dedicated point-to-point link → no bus contention
  • Gyro data arrives continuously via UART — good for streaming sensor data
  • I2C for memory allows random access read/write with address control
  • Two separate peripherals, each using its strongest interface

3) Selection: I2C for Gyro

  • I2C standard mode = 100kbps, fast mode = 400kbps >> Gyro UART speed of 9600bps
  • Fewer pins used overall (both on same 2-wire bus)
  • UART requires 2 additional pins (TX, RX) and separate wiring
  • I2C supports multi-master if needed in future

               └──── SCL ──── Gyro MEMS

Q13
Stepper Motor — Two-Coil Excitation, 4-Step Sequence
May 2023 · Dec 2023 · Jan-May 2024 6 marks
▾
// Question
Stepper motor in two-coil excitation (high torque, two phase). Coils connected via UNLN2003 Darlington array to Port 1 higher nibble (P1.7, P1.6, P1.5, P1.4). Write C/assembly to initialise ports and rotate motor 100 times through 4-step sequence: 1001 → 0011 → 0110 → 1100 (on Port1 higher nibble), in same order.
// Answer

The higher nibble of P1 (bits 7–4) drives the stepper coils. We mask out the lower nibble each time to preserve those bits.

#include "msp430g2553.h"

// Stepper sequence on upper nibble: P1.7 P1.6 P1.5 P1.4
// Sequence bits (4-bit): 1001, 0011, 0110, 1100
// Shifted to upper nibble (×16): 0x90, 0x30, 0x60, 0xC0

const unsigned char step_seq[4] = {0x90, 0x30, 0x60, 0xC0};

void delay_ms(unsigned int ms) {
    unsigned int i;
    for (i = 0; i < ms; i++)
        __delay_cycles(1000); // 1MHz: 1000 cycles = 1ms
}

int main(void) {
    WDTCTL = WDTPW + WDTHOLD;

    // Initialise P1.7–P1.4 as outputs
    P1DIR |= (BIT7 | BIT6 | BIT5 | BIT4);
    P1OUT &= ~(BIT7 | BIT6 | BIT5 | BIT4); // all OFF

    int rotation, step;

    for (rotation = 0; rotation < 100; rotation++) {
        for (step = 0; step < 4; step++) {
            // Clear upper nibble, set new step
            P1OUT = (P1OUT & 0x0F) | step_seq[step];
            delay_ms(10); // step delay
        }
    }

    // Turn off coils after done
    P1OUT &= 0x0F;
    return 0;
}
Step Sequence Breakdown
P1.7 P1.6 P1.5 P1.4 → hex (upper nibble)
1 0 0 1 → 0x90  |  0 0 1 1 → 0x30  |  0 1 1 0 → 0x60  |  1 1 0 0 → 0xC0
